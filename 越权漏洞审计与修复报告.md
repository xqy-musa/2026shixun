## 一、实验概述

### 1.1 实验背景

本次实验对「用户信息管理平台」中新增的个人中心与充值功能进行了全面的安全审计与修复。越权漏洞（Privilege Escalation）是 Web 应用中最常见的安全缺陷之一，其本质是系统未能有效验证用户对其请求资源的操作权限，导致普通用户可以访问或操作本应受限的数据与功能。

### 1.2 审计范围

| 项目 | 内容 |
|------|------|
| 审计对象 | `/opt/Class01/app.py` 中的 `/profile`、`/recharge` 路由 |
| 存储数据 | `data/users.db` — users 表（id, username, email, phone, balance） |
| 审计内容 | 身份验证、权限校验、数据访问控制、敏感信息展示 |
| 审计方法 | 白盒代码审计 + 黑盒渗透测试 |
| 审计工具 | Burp Suite + Python 命令行验证 |

### 1.3 测试环境准备

```bash
# 1. 启动应用
cd /opt/Class01 && python3 app.py

# 2. 浏览器访问
# 管理员：admin / admin123

# http://127.0.0.1:5000/profile
# POST http://127.0.0.1:5000/recharge
```

## 三、漏洞详细分析

---

### PRIV-01 水平越权 — 未授权访问任意用户资料

#### 漏洞描述

个人中心接口 `/profile` 通过 URL 查询参数 `user_id` 指定要查询的用户。由于服务端未对当前会话用户与所查询 `user_id` 的一致性进行校验，任意登录用户可通过修改 URL 参数遍历查看系统中所有用户的个人资料。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`GET /profile`
- **代码行号**：修复前第 324 行

#### 漏洞代码（修复前）

```python
# user_id 来自 URL 参数（客户端完全可控）
user_id = request.args.get("user_id")

# ❌ 没有验证当前登录用户和要查询的 user_id 是否匹配
c.execute("SELECT id, username, email, phone, balance FROM users WHERE id = ?", (user_id,))
user_data = c.fetchone()
return render_template("profile.html", user=dict(user_data))
```

#### 漏洞可利用的 Payload

```
GET /profile?user_id=1    ← 修改数字即可查看任意用户
```

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：以 alice 登录系统
        浏览器访问 http://127.0.0.1:5000/login
        使用 alice / alice2025 登录
──────────────────────────────────────────────────────────────────
步骤 2：打开 Burp Suite，配置浏览器代理
        Proxy → Intercept → Intercept is on
──────────────────────────────────────────────────────────────────
步骤 3：浏览器访问 http://127.0.0.1:5000/profile?user_id=1
        Burp Suite 拦截到 GET 请求：

        GET /profile?user_id=1 HTTP/1.1
        Host: 127.0.0.1:5000
──────────────────────────────────────────────────────────────────
步骤 4：放行请求（Forward），查看响应内容
──────────────────────────────────────────────────────────────────
步骤 5：响应中显示了 user_id=1（admin）的用户资料
        包括邮箱、手机号、余额等隐私信息
        ⚠️ 当前登录用户是 alice，但成功查看了 admin 的资料
──────────────────────────────────────────────────────────────────
步骤 6：继续修改 user_id=3、4、5... 遍历所有用户
        ✅ 批量获取全部用户数据
──────────────────────────────────────────────────────────────────
```

#### 测试结果

| 测试 Payload | 查看目标 | 预期 | 实际 |
|-------------|----------|:----:|:----:|
| `/profile?user_id=1` | admin 资料 | 拒绝访问 | ❌ 查看成功 |
| `/profile?user_id=2` | alice 资料 | 允许访问 | ✅ 查看成功 |
| `/profile?user_id=3` | 其他用户资料 | 拒绝访问 | ❌ 查看成功 |

#### 可能造成的危害

- 任意用户隐私信息泄露（邮箱、手机号等）
- 余额信息暴露，可用于社会工程学攻击
- 批量遍历获取全部用户数据

### PRIV-02 垂直越权 — 跨用户充值/扣款

#### 漏洞描述

充值接口 `/recharge` 从表单隐藏字段获取目标用户的 `user_id`，攻击者可通过 Burp Suite 拦截 POST 请求修改该参数，对任意用户的账户余额执行充值或扣款操作。该漏洞使普通用户能够越权操作管理员账户。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /recharge`
- **代码行号**：修复前第 347 行

#### 漏洞代码（修复前）

```python
# user_id 来自表单隐藏字段（客户端可控）
user_id = request.form.get("user_id")
amount = request.form.get("amount", "0")

# ❌ 没有验证当前登录用户和操作目标是否匹配
# ❌ 没有校验 amount 正负
new_balance = row[0] + float(amount)
c.execute("UPDATE users SET balance = ? WHERE id = ?", (new_balance, user_id))
```

#### 漏洞可利用的 Payload

```
# 给 admin 扣光余额
POST /recharge
user_id=1&amount=-99999
```

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：以 alice 登录，打开个人中心 /profile
──────────────────────────────────────────────────────────────────
步骤 2：Burp Suite 开启拦截（Intercept is on）
──────────────────────────────────────────────────────────────────
步骤 3：在充值表单输入金额 100，点击充值按钮
──────────────────────────────────────────────────────────────────
步骤 4：查看拦截到的 POST 请求内容

        POST /recharge HTTP/1.1
        Host: 127.0.0.1:5000

        csrf_token=xxx&user_id=2&amount=100
──────────────────────────────────────────────────────────────────
步骤 5：将请求修改为：

        csrf_token=xxx&user_id=1&amount=-99999
──────────────────────────────────────────────────────────────────
步骤 6：放行请求（Forward）
──────────────────────────────────────────────────────────────────
步骤 7：以 admin 登录，查看余额
        结果：余额从 ¥99,999 变为 ¥0
        ⚠️ 一个普通用户扣光了管理员的全部余额！
──────────────────────────────────────────────────────────────────
```

#### 测试结果

| 测试场景 | 修改内容 | 实际结果 |
|----------|----------|:--------:|
| 给自己加钱 | user_id=2, amount=9999999 | ❌ alice 余额暴涨 |
| 给别人扣钱 | user_id=1, amount=-99999 | ❌ admin 余额被扣光 |
| 改别人账号操作 | user_id=3, amount=-5000 | ❌ 任意用户余额可改 |

#### 可能造成的危害

- 任意用户余额可被他人篡改
- 管理员余额可被普通用户扣光
- 系统经济体系完全失控

### PRIV-03 金额参数负值未校验

#### 漏洞描述

充值接口对金额参数 `amount` 未做正负校验，允许提交负数。这导致"充值"接口实际上可被用作"扣款"接口，结合 PRIV-02 可对任意用户执行余额扣减。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /recharge`
- **代码行号**：修复前第 348 行

#### 漏洞代码（修复前）

```python
# ❌ 没有校验 amount 是否为正数
amount = request.form.get("amount", "0")
new_balance = row[0] + float(amount)    # 负数直接累加
```

#### 漏洞可利用的 Payload

```
amount=-99999    ← 填负数实现扣款
amount=0         ← 填零无意义
amount=-1        ← 填 -1 每次扣 1 元
```

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：登录后，Burp Suite 拦截充值 POST 请求
──────────────────────────────────────────────────────────────────
步骤 2：将 amount 字段改为负数，如 -99999
──────────────────────────────────────────────────────────────────
步骤 3：放行请求，服务端直接执行：
        new_balance = 100 + (-99999) = -99899
──────────────────────────────────────────────────────────────────
步骤 4：刷新页面查看余额
        结果：余额从 ¥100 变为 ¥-99899 ⚠️
──────────────────────────────────────────────────────────────────
```

#### 测试结果

| 测试输入 | 期望 | 实际 | 判定 |
|----------|:----:|:----:|:----:|
| amount=-100 | 拒绝充值 | 余额减少 100 | ❌ 漏洞存在 |
| amount=0 | 拒绝充值 | 余额不变 | ⚠️ 逻辑缺陷 |
| amount=-99999 | 拒绝充值 | 余额减少 99999 | ❌ 可扣光 |

#### 可能造成的危害

- 余额可被扣为负数
- 资金被窃取
- 系统数据被破坏

### PRIV-04 敏感信息未脱敏展示

#### 漏洞描述

个人中心页面未对手机号、邮箱、余额等敏感信息进行脱敏处理，直接展示数据库原始数据。而首页已实现了完整的脱敏逻辑，存在安全处理标准不一致的问题。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`GET /profile`
- **代码行号**：修复前第 337 行

#### 漏洞代码（修复前）

```python
# ❌ 没有调用脱敏函数，原始数据直接传入模板
return render_template("profile.html", user=dict(user_data))

# 数据库中原始数据：
# phone = "13800138000"     完整手机号
# balance = 99999           原始数值
```

#### 对比：首页已实现的脱敏逻辑

```python
def sanitize_user_info(user_info):
    return {
        "phone": user_info["phone"][:3] + "****" + user_info["phone"][-4:],
        "balance": "¥{:,.2f}".format(float(user_info["balance"])),
    }

# 首页显示效果：
# phone = "138****8000"
# balance = "¥99,999.00"
```

#### 可能造成的危害

- 用户手机号、邮箱等隐私信息完全暴露
- 截图或共用电脑时造成信息泄露
- 不符合数据最小化展示原则

## 四、修复方案

### 4.1 修复策略总览

| 漏洞编号 | 修复策略 | 具体措施 |
|----------|----------|----------|
| PRIV-01 | 身份来源切换 | user_id 改为从 session 获取，不信任 URL 参数 |
| PRIV-02 | 操作对象锁定 | 充值仅操作当前登录用户，不从表单获取 user_id |
| PRIV-03 | 参数业务校验 | amount 必须 > 0，否则拒绝并提示 |
| PRIV-04 | 统一脱敏逻辑 | 个人中心复用脱敏处理，与首页标准一致 |

### 4.2 各漏洞详细修复

#### PRIV-01 修复：身份来源切换

**修复原理**：将用户身份标识来源从 URL 参数切换为服务端 session，消除客户端对身份参数的控制权。

```python
# 【修复前】从 URL 参数获取
user_id = request.args.get("user_id")
c.execute("SELECT ... FROM users WHERE id = ?", (user_id,))

# 【修复后】从 session 获取当前登录用户名
username = session.get("username")
c.execute("SELECT ... FROM users WHERE username = ?", (username,))
```

#### PRIV-02 修复：操作对象锁定

**修复原理**：充值不再接收前端传来的 user_id，直接从 session 确定操作对象。

```python
# 【修复前】从表单隐藏字段获取 user_id
user_id = request.form.get("user_id")

# 【修复后】从 session 获取当前登录用户
username = session.get("username")
c.execute("UPDATE users SET balance = ? WHERE username = ?", (new_balance, username))
```

#### PRIV-03 修复：金额正负校验

**修复原理**：对 amount 进行严格的正数校验，负数和零直接拒绝，同时捕获非法输入。

```python
# 【修复前】直接累加
new_balance = row[0] + float(amount)

# 【修复后】三层校验
try:
    amount_float = float(amount)
    if amount_float <= 0:
        return render_template("profile.html", error="充值金额必须大于 0")
except ValueError:
    return render_template("profile.html", error="请输入有效的金额")
new_balance = row[0] + amount_float
```

#### PRIV-04 修复：数据脱敏

**修复原理**：复用首页已有的脱敏逻辑，确保个人中心与首页数据展示标准一致。

```python
# 【修复前】原始数据直接展示
return render_template("profile.html", user=dict(user_data))

# 【修复后】脱敏处理后展示
safe_info = {
    "id": user_dict["id"],
    "username": user_dict["username"],
    "email": user_dict["email"],
    "phone": user_dict["phone"][:3] + "****" + user_dict["phone"][-4:],
    "balance": "¥{:,.2f}".format(float(user_dict["balance"])),
}
return render_template("profile.html", user=safe_info)
```

### 4.3 修复前后代码对比

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| 身份标识来源 | URL参数 / 表单字段（客户端可控） | session（服务端控制） |
| 跨用户权限校验 | ❌ 无 | ✅ 仅操作当前用户 |
| 金额正负校验 | ❌ 无 | ✅ 必须 > 0 |
| 敏感信息脱敏 | ❌ 原始数据展示 | ✅ 手机号脱敏 + 余额格式化 |

## 六、总结与建议

### 6.1 修复成果

本次越权安全专项审计共发现 **4 项漏洞**（高危 3 项、中危 1 项），已全部完成修复。经 Burp Suite 验证，所有越权攻击均被有效拦截，系统权限控制目前已处于安全状态。

### 6.2 根本原因分析

| 问题根源 | 说明 | 涉及漏洞 |
|----------|------|----------|
| 服务端过度信任客户端 | user_id 完全由 URL/表单参数决定 | PRIV-01, PRIV-02 |
| 输入校验缺失 | amount 未做业务逻辑约束 | PRIV-03 |
| 安全编码标准不统一 | 首页有脱敏，个人中心没有 | PRIV-04 |

### 6.3 安全编码建议

在开发涉及用户数据访问和资金操作的接口时，请务必遵循以下原则：

1. **永远不要信任前端传来的用户标识**
   - user_id 等身份标识必须从服务端 session 获取
   - 不能来自 URL 参数或表单隐藏字段

2. **每个接口都要做权限校验**
   - 每次操作前校验"当前用户是否有权执行此操作"
   - 不能依赖前端做权限控制

3. **敏感数据统一脱敏**
   - 所有展示用户信息的页面使用同一套脱敏逻辑
   - 避免安全标准不一致

4. **金额操作严格校验**
   - 涉及金额的操作必须校验正负、上限
   - 在服务端记录操作日志

### 6.4 后续优化建议

|--------|--------|------|
| 🟡 中 | 操作日志审计 | 记录每次充值的时间、用户、IP、金额，便于追溯 |
