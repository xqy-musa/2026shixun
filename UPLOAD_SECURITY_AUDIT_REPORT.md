## 1. 实验概述

### 1.1 实验背景

本次安全审计针对「用户信息管理平台」中新增加的头像上传功能进行专项检测。文件上传功能是 Web 应用中最容易被攻击者利用的功能之一，一旦存在安全缺陷，攻击者可以直接上传恶意脚本文件（如 Webshell）控制服务器。

### 1.2 审计范围

| 项目 | 内容 |
|------|------|
| 审计对象 | `/opt/Class01/app.py` 中的 `/upload` 路由 |
| 存储目录 | `static/uploads/` |
| 审计内容 | 文件类型校验、路径安全、文件命名、文件内容校验 |
| 模拟攻击 | Webshell 上传、路径遍历、文件覆盖 |

### 1.3 测试环境准备

```bash
# 1. 启动应用
cd /opt/Class01 && python3 app.py

# 2. 浏览器访问（先注册或登录）
# 管理员：admin / admin123
# 普通用户：alice / alice2025

# 3. 登录后访问上传页面
# http://127.0.0.1:5000/upload
```

## 3. 漏洞详细分析与复现

---

### UPL-01 任意文件上传（Webshell 上传）

#### 📌 漏洞描述

上传接口**没有对文件类型做任何限制**，攻击者可以上传任意后缀的文件（如 `.php`、`.py`、`.asp`、`.exe`），并直接通过浏览器访问执行。

#### 🔍 漏洞代码位置

**文件**：`app.py` | **路由**：`POST /upload` | **修复前第 278 行**

```python
# ========== 缺少的关键校验 ==========
# ❌ 没有检查文件后缀（.php .py .asp ...）
# ❌ 没有检查 MIME Content-Type
# ❌ 没有检查文件内容是否为真实图片
```

#### 💣 漏洞 Payload

```php
<?php system($_GET['cmd']); ?>
```

保存为 `shell.php`，通过上传功能提交后访问：

```
http://127.0.0.1:5000/static/uploads/shell.php?cmd=whoami
```

#### 📷 复现步骤（BurpSuite + 客户视角）

```

│  使用 admin / admin123 登录                                │
├────────────────────────────────────────────────────────────┤
│  步骤 2：打开 BurpSuite，配置代理拦截                       │
│  Proxy → Intercept → Intercept is on                      │
├────────────────────────────────────────────────────────────┤
│  步骤 3：访问上传页面 http://127.0.0.1:5000/upload         │
│  选择 webshell.php 文件上传，BurpSuite 拦截到请求           │
├────────────────────────────────────────────────────────────┤
```

| `恶意代码` | `.exe` | ✅ 上传成功 | ✅ 文件可下载 |

#### ⚠️ 可能造成的危害

- 攻击者可上传 Webshell，**完全控制服务器**
- 执行任意系统命令（`ls`、`whoami`、`cat /etc/passwd`）
- 读取、修改、删除服务器上的任意文件
- 以服务器为跳板攻击内网其他系统

### UPL-02 路径遍历漏洞

#### 📌 漏洞描述

上传接口直接将用户传入的**原始文件名**用于文件保存路径，攻击者可通过文件名中的 `../` 将文件写入服务器上的任意目录。

#### 🔍 漏洞代码位置

**文件**：`app.py` | **修复前第 279 行**

```python
# ❌ 用户传入 ../../tmp/evil.php，保存路径变为：
# static/uploads/../../tmp/evil.php = /opt/Class01/tmp/evil.php
filepath = os.path.join(upload_dir, file.filename)
```

#### 💣 漏洞 Payload

将文件命名为：

```
../../../tmp/evil.php
```

保存后的实际路径为 `/opt/Class01/tmp/evil.php`，实现了跨目录写入。

#### 📷 复现步骤

```
┌────────────────────────────────────────────────────────────┐
│  步骤 1：登录后访问 /upload                                │
├────────────────────────────────────────────────────────────┤
│  步骤 2：使用 BurpSuite 拦截上传请求                       │
│  修改 filename 为：                                       │
│  Content-Disposition: form-data; name="file";             │
│                    filename="../../../tmp/hacked.txt"      │
├────────────────────────────────────────────────────────────┤
│  步骤 3：放行请求                                          │
│                                                           │
│  结果验证：                                                │
│  ls -la /tmp/hacked.txt  → 文件存在于 /tmp/ 目录          │
│  ls -la static/uploads/  → 该目录下没有这个文件            │
└────────────────────────────────────────────────────────────┘
```

#### ⚠️ 可能造成的危害

- 覆盖系统关键文件（如 `/etc/passwd`、`/etc/crontab`）
- 将恶意文件写入 Web 根目录之外的自启动目录
- 绕过目录权限限制，写入本不应可写的目录

### UPL-03 文件名冲突导致文件覆盖

#### 📌 漏洞描述

上传时**未做文件名去重**，用户多次上传同名文件，后上传的文件会直接覆盖之前已上传的文件，造成数据丢失。

#### 🔍 漏洞代码位置

**文件**：`app.py` | **修复前第 279 行**

```python
# ❌ 两次上传 avatar.jpg，第一次的文件被覆盖
filepath = os.path.join(upload_dir, file.filename)
file.save(filepath)
```

#### 📷 复现步骤

```
┌────────────────────────────────────────────────────────────┐
│  步骤 1：上传 avatar.jpg（内容为红色图片）                  │
│  步骤 2：上传 avatar.jpg（内容为蓝色图片）                  │
│  步骤 3：访问 /static/uploads/avatar.jpg                   │
│                                                           │
│  结果：显示的是蓝色图片 ❌ 红色图片已被覆盖                  │
└────────────────────────────────────────────────────────────┘
```

#### ⚠️ 可能造成的危害

- 用户头像被他人同名文件覆盖
- 重要数据因同名上传而丢失
- 恶意用户可上传同名文件替换已有合法文件

---

### UPL-04 缺少文件内容真实性校验

#### 📌 漏洞描述

接口**不校验上传文件的内容是否为真实图片**。即使文件后缀为 `.jpg`，内容也可以是任意格式（如 PHP 代码），通过修改后缀即可绕过简单的后缀检查。

#### 🔍 漏洞代码位置

**文件**：`app.py` | **所有上传处理逻辑**

```python
# ❌ 没有读取文件头判断是否为真实图片
# ❌ 没有使用 PIL/Pillow 打开验证
# ❌ 直接保存文件内容
file.save(filepath)
```

#### 💣 漏洞 Payload

```
filename="shell.jpg"
Content-Type: image/jpeg
文件内容：
<?php system($_GET['cmd']); ?>
```

#### ⚠️ 可能造成的危害

- 虽然扩展名是 `.jpg`，但如果服务器配置了双后缀解析（Apache 的 `AddHandler`），`shell.jpg` 仍可能被当 PHP 执行
- 图片马绕过前端 JS 校验
- 被用于结合其他漏洞进行组合攻击

## 4. 修复方案

### 修复策略总览

本次修复采用 **白名单 + 路径净化 + UUID 重命名 + 内容验证** 的四层防御策略。

```
┌─────────────────────────────────────────────────┐
│                  文件上传请求                     │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  第1层：扩展名白名单校验          UPL-01 ✅      │
│  仅允许：jpg, jpeg, png, gif, webp              │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  第2层：路径遍历防护              UPL-02 ✅      │
│  os.path.basename() 提取纯粹文件名              │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  第3层：UUID 重命名                UPL-03 ✅      │
│  uuid4().hex 生成唯一文件名，防止覆盖             │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  第4层：文件内容真实性校验        UPL-04 ✅      │
│  imghdr.what() 检查文件头是否为真实图片           │
└─────────────────────────────────────────────────┘
```

### 各漏洞详细修复代码

#### 🛠 UPL-01 修复：扩展名白名单

```python
# 【新增】白名单配置
ALLOWED_EXTENSIONS = {"jpg", "jpeg", "png", "gif", "webp"}

def allowed_file(filename):
    """检查文件扩展名是否在白名单内。"""
    return "." in filename and filename.rsplit(".", 1)[1].lower() in ALLOWED_EXTENSIONS

# 【新增】在 upload 路由中调用校验
if not allowed_file(file.filename):
    return render_template("upload.html", error="仅允许上传 jpg、jpeg、png、gif、webp 格式的图片文件")
```

#### 🛠 UPL-02 修复：路径遍历防护

```python
# 【修复】使用 os.path.basename() 提取纯粹文件名
# 输入: ../../etc/passwd  → 输出: passwd
# 输入: ../../../tmp/x.php → 输出: x.php
original_name = os.path.basename(file.filename)
```

#### 🛠 UPL-03 修复：UUID 重命名

```python
# 【修复】使用 UUID 生成唯一文件名
import uuid

ext = original_name.rsplit(".", 1)[1].lower() if "." in original_name else ""
safe_filename = f"{uuid.uuid4().hex}.{ext}"
# 示例输出: a3f8c91e2b4d5f6a7c8b9d0e1f2a3b4c.jpg
```

#### 🛠 UPL-04 修复：文件内容真实性校验

```python
# 【修复】使用 imghdr 检查文件头是否为真实图片
import imghdr  # 无需 pip 安装，Python 标准库

if not imghdr.what(filepath):
    os.remove(filepath)
    return render_template("upload.html", error="上传的文件不是有效的图片格式")
```

### 修复前后代码对比

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| 文件后缀检查 | ❌ 无 | ✅ 白名单：jpg/jpeg/png/gif/webp |
| 路径遍历防护 | ❌ 无 | ✅ `os.path.basename()` 提取文件名 |
| 文件重命名 | ❌ 原始文件名 | ✅ UUID 重命名 |
| 文件类型验证 | ❌ 无 | ✅ `imghdr.what()` |
| 文件大小限制 | 16MB | 16MB（不变，已足够） |

## 6. 总结与建议

### 6.1 修复成果

本次文件上传安全专项审计共发现 **4 项漏洞**（高危 2 项、中危 2 项），已全部完成修复并验证通过。所有 Webshell 上传、路径遍历、文件覆盖攻击均被有效拦截。

### 6.2 根本原因分析

| 问题根源 | 说明 | 涉及漏洞 |
|----------|------|----------|
| **缺少输入验证** | 未对用户上传的文件类型做任何检查 | UPL-01, UPL-04 |
| **信任用户输入** | 直接将用户提供的文件名用于路径拼接 | UPL-02 |
| **缺少唯一性处理** | 未处理文件名冲突 | UPL-03 |

### 6.3 文件上传安全开发规范（必读）

在开发文件上传功能时，请务必遵循以下 **"四个必须"**：

```
┌──────────────────────────────────────────────────────────┐
│  ① 必须做扩展名白名单校验                                  │
│     ✅ 仅允许需要的类型，禁止一切未明确允许的类型             │
│                                                           │
│  ② 必须做路径遍历防护                                      │
│     ✅ os.path.basename() 取出纯粹文件名                   │
│                                                           │
│  ③ 必须做文件重命名                                        │
│     ✅ UUID / 时间戳，拒绝使用用户提供的原始文件名            │
│                                                           │
│  ④ 必须做文件内容校验                                      │
│     ✅ 读取文件头魔术字节 / 使用专用库打开验证                │
└──────────────────────────────────────────────────────────┘
```

### 6.4 补充建议

| 优先级 | 建议项 | 说明 |
|--------|--------|------|
| 🟢 建议 | 上传目录添加 `.htaccess` 或 Nginx 规则 | 禁止执行 uploads 目录中的脚本文件 |
| 🟢 建议 | 限制单用户上传频率 | 防止恶意批量上传造成磁盘空间耗尽 |
| 🟢 建议 | 设置合理的文件大小上限 | 头像建议 ≤ 2MB，当前 16MB 可适当调小 |
